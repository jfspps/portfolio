Secure Student-Record-Management (Secure-SRM)

Overview

SRM is an academic database which can record student results and provide school staff with basic contact details for all personnel (teachers and guardians) who are related to each student. Secure-SRM is the Spring Security backed application.

Motivation

Following from my experience with BodyBand, I spent a few days planning the project before any coding, and outlined my ideas as a former-teacher, writing a Systems requirement specification. As you will see, I previously referred to the project as 'Linq' but then changed it to SRM.

Docker image demo

A Docker image is available here.

The project uses port 5000 by default and can be mapped to the host's port by changing the value for xxxx:

docker run -d -p xxxx:5000 jfspps/secure-srm

Open a browser and go to "localhost:xxxx" to login. Use any of the three user types (each with different access levels):

  1. Admin user username: "amysmith", password: "amysmith123"
  2. Teacher user username: "marymanning", password: "marymanning123"
  3. Guardian user username: "alexsmith", password: "alexsmith123"

In general, Admin users are able to add, remove or update Teacher and Guardian accounts. Admin users cannot edit student results. Teacher users can only edit the results of their own students and not of other teachers. Guardians can only view the results of their own dependents.

Web-login: authentication and authorisation

The Spring Security frontend is based on a separate project, Web-login. Secure-SRM is a combination of SRM and Web-login. The list of key features include:

  • Bootstrap (CSS) and Thymeleaf front-end
  • Granular permissions. Please see BootLoader:
    • Authority entity (a list of Users with the given authority) and Roles (ADMIN, USER, etc.)
    • User entity, with a list of Authority's (a user may belong to more than one Authority)
    • Role entity, with a list of Users (maps a role name ADMIN or USER etc., to a group of users)
  • Password encryption technologies e.g. BCrypt and Cross-Site Request Forgery (CSRF) protection
  • Custom Spring annotations, with @interface, that handle routing authentication
  • User login/logout timestamps and 403 (forbidden) responses and other user account properties (lockout, expired, modified)
  • In-memory H2 and MySQL persistent user accounts and Remember-Me tokens
  • Example database entity (known as TestRecord) to demonstrate authorised CRUD operations of Web-login
  • Automatic account lock-out after repeated failed login attempts via listener classes

The construction of Secure-SRM proceeded by replacing the TestRecord entity with database entities that are relevant to SRM.

MySQL database

At this stage I had just finished an online MySQL course and was familiar enough with MySQL 8 and MySQL workbench to begin planning. I decided to build a NodeJS and MySQL based parent's portal for SRM which also contains a lot of the preliminary planning for Secure-SRM. In the end it turned out to be a substantial project, trying to grapple with Datatables. Originally, I had planned to build the entire project with NodeJS instead of Spring but then decided to build on my Java skills and get some experience with Spring 5. The project presented in this summary is the Spring Security based project.

Part of my planning was also carried out using MySQL workbench (part of the original schema is reproduced above).
